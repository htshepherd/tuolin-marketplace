## 类型说明

- **AFK**：需求和验收标准已经明确，可由开发 Agent 独立实现、测试和提交。
- **HITL**：实现过程中存在必须由人确认的质量判断或真实外部能力结论。

## 切片清单

### 1. S01 — 建立可信的 YouTube-only 视频任务入口

- **类型**：AFK
- **Blocked by**：无，可立即开始
- **覆盖用户故事**：1–6、18

#### What to build

交付一条可演示的任务创建路径：用户用自然语言提出 YouTube Shorts 产品视频需求后，系统创建独立运行目录，只从已发布的知识库 Agent 读取接口加载匹配产品的正式知识卡、官方图片和有效授权视频档案，并生成可继续追问的初始状态。TikTok、缺失或过期接口、产品身份不一致、非正式素材、stale/revoked 视频及未支持事实必须在任务入口处明确阻断。

#### Acceptance criteria

- [ ] 自然语言创建一个 YouTube Shorts 视频任务时，会产生独立运行目录和可恢复状态。
- [ ] 初始状态记录产品身份、知识版本、正式知识来源、官方图片清单、有效视频档案目录能力、语言和目标时长。
- [ ] 同一请求重复创建时不会复用或污染旧运行目录。
- [ ] TikTok-only、YouTube 加 TikTok 或其他未支持平台均返回明确的当前版本边界，不创建可继续执行的错误任务。
- [ ] Agent 读取接口缺失、校验失败、产品不匹配或已知过期时，视频任务被标记为 blocked，不能进入追问。
- [ ] 产品名称、参数、认证和应用只能来自正式知识上下文；初始请求中的冲突信息不会覆盖正式身份。
- [ ] Codex 回复直接说明当前阶段、已读取的知识来源、官方图片数量、可用视频档案数量、阻塞项和下一步自然语言动作。
- [ ] 源 Skill、插件镜像、自然语言路由和外部行为测试表现一致。

### 2. S02 — 用动态创意追问与决策账本替代固定六问

- **类型**：HITL
- **Blocked by**：S01
- **覆盖用户故事**：7–17、30–31、75

#### What to build

交付 Creative Discovery Core 的第一条完整路径：系统从初始请求和正式证据中识别已知决策，针对当前最高价值的未决业务问题提出一个有理由的建议，在 Codex 中只询问这一项；用户可以普通确认或直接纠正。决策账本记录建议、证据、确认、纠正、历史和剩余决策，并以 Decision Sufficiency 而不是固定问题数判断能否生成 Confirmed Video Brief。

#### Acceptance criteria

- [ ] 不再以固定字段顺序、固定六问或预设创意方向菜单驱动访谈。
- [ ] 每轮只显示一个业务问题、一个建议和一个理由，并以“是否确认”结束。
- [ ] 用户回复普通“确认”只确认当前建议，不授权其他未决项或后续阶段。
- [ ] 用户直接给出修正答案时，账本保存修正内容、来源和历史，并据此选择下一项依赖决策。
- [ ] 已在初始请求或正式知识中明确的信息不会重复询问。
- [ ] 模糊受众、空泛观看理由、纯炫技方向和无业务价值建议会被明确质疑，而不是直接接受。
- [ ] 追问能够收敛 Actionable Video Audience、Audience Problem Scenario、观看/继续观看理由、Viewer-Interest Direction、记忆点、期望行动、优先事实、Human-Relevance Angle、排除项、声明风险和 AI 模拟边界。
- [ ] Human-Relevance Angle 通过具体人物情境判断，不使用抽象的“生活化/专业化”二选一。
- [ ] 只有满足语义完整性且无 blocker 时才能生成 Confirmed Video Brief；固定回答数量不能绕过该门禁。
- [ ] Creative Discovery Core 可用纯数据输入输出独立测试，不依赖 Dreamina、文件展示或付费操作。
- [ ] HITL 验收使用至少一个真实工业品请求走完整段访谈，由用户确认其问题顺序、质疑力度和建议质量达到预期。

### 3. S03 — 将公开 YouTube 趋势证据接入观众兴趣方向

- **类型**：AFK
- **Blocked by**：S02
- **覆盖用户故事**：19–25

#### What to build

交付一条趋势证据闭环：当受众和问题场景明确后，Agent 必须在确认最终 Viewer-Interest Direction 前扫描目标语言和地区的公开 YouTube 案例，按相关性阶梯提取可迁移的短视频机制、排除不适用手法，并把含来源和日期的精简 Trend Evidence Brief 直接展示在 Codex 中。

#### Acceptance criteria

- [ ] 趋势扫描只能在 Actionable Video Audience 和 Audience Problem Scenario 已确认后触发。
- [ ] 搜索使用目标观众语言和地区，而不是默认使用中文操作者语言。
- [ ] 当前版本只接受公开 YouTube 信号，不写入 TikTok 或虚构的第一方频道分析。
- [ ] 案例选择依次遵守“同类工业品 → 相邻制造/工程 → 可迁移大众机制”的相关性阶梯。
- [ ] 每条采用的证据记录来源链接、扫描日期、相关层级、共同有效原因和可迁移机制。
- [ ] Trend Evidence Brief 在 Codex 中显示采用机制、排除手法、证据强度和来源，不要求用户打开研究文件。
- [ ] 无法获得当前数据时明确标为降级推理，不把模型推断伪装成已验证趋势。
- [ ] 不适合目标受众、违反正式事实或带来宣传风险的热门手法会被排除并说明原因。
- [ ] 缺少合格 Trend Evidence Brief 时，最终 Viewer-Interest Direction 不能确认，策划不能生成。
- [ ] 测试通过可控证据夹具验证排序、降级、排除和门禁，不依赖不稳定的实时热度数值。

### 4. S04 — 在方向确认前建立图片素材可行性门禁

- **类型**：AFK
- **Blocked by**：S02
- **覆盖用户故事**：26–29

#### What to build

交付一条素材证据闭环：Agent 在确认 Viewer-Interest Direction 前，必须实际打开相关官方图片并记录主体、清晰度、构图、9:16 裁切、近重复和创意支持能力。素材结论进入决策账本；不重复可用图片不足时，用户只能缩短时长或明确批准有意重复。

#### Acceptance criteria

- [ ] 每张候选官方图片都有完整像素检查记录，不能仅根据文件名、标签或路径推断。
- [ ] 检查结果至少覆盖主体、清晰度、构图、竖屏裁切、近重复和支持的画面用途。
- [ ] 近重复图片被归组，不会被重复计入独立素材容量。
- [ ] 素材检查结果能明确回答当前 Viewer-Interest Direction 是否可执行，而不从像素中推导新产品事实。
- [ ] 不重复素材容量不足时，计划确认被阻止，并在 Codex 中显示支持的时长及原因。
- [ ] 用户缩短时长后，实际运行状态、预估镜头容量和后续展示同步更新。
- [ ] 用户明确批准有意重复后，批准范围和原因写入决策账本，不能被默认推断。
- [ ] 未覆盖所有候选图、路径不可读或检查 JSON 异常时明确 blocked。
- [ ] 素材检查是 Codex 内部动作，不要求用户运行脚本或编辑检查文件。
- [ ] 测试覆盖可用素材、近重复、容量不足、缩短时长和有意重复批准的完整用户路径。

### 5. S05 — 将充分决策自动转换为可在 Codex 中确认的策划

- **类型**：AFK
- **Blocked by**：S03、S04
- **覆盖用户故事**：32–37、70

#### What to build

交付从 Decision Sufficiency 到策划确认的完整路径：Confirmed Video Brief 自动生成正式策划，Codex 直接显示受众问题、观众兴趣方向、趋势证据、叙事、产品事实、风险、时长和 3–6 张已检查代表图片。用户可以用自然语言修改允许的策划字段，普通确认只锁定当前策划并进入分镜阶段。

#### Acceptance criteria

- [ ] 只有决策充分、趋势证据完成且素材可行时才能生成正式策划。
- [ ] 策划只消费 Confirmed Video Brief，不重新猜测或覆盖已确认的访谈决策。
- [ ] Codex 策划确认视图直接显示受众问题、Viewer-Interest Direction、趋势机制与来源、核心叙事、优先事实、风险、时长和代表图片。
- [ ] 代表图片为已完成像素检查的 3–6 张素材，并直接显示而非只列路径。
- [ ] 中文操作者可以用中文审阅英文视频策划，但外部产品名和目标语言内容不会被中文标签污染。
- [ ] 自然语言语义修改会实际更新允许字段、重新显示结果并记录变更历史。
- [ ] 产品身份、知识来源、正式事实、平台、官方素材边界和确认门禁不能通过语义修改覆盖。
- [ ] 普通“确认”只确认当前策划；没有生成策划或策划已因修改失效时不能确认。
- [ ] 阶段回执说明产物、验证项、warning/blocker 和下一步自然语言动作。
- [ ] 外部行为测试覆盖成功生成、门禁阻断、语义修改、受保护字段和阶段确认。

### 6. S06 — 从确认策划生成动态可视化分镜并支持自然语言修订

- **类型**：AFK
- **Blocked by**：S05
- **覆盖用户故事**：38–39、42、44–45

#### What to build

交付一条可独立演示的分镜路径：确认策划后，系统根据叙事和素材动态决定镜头数量、时间、节奏和顺序。图片镜头继续支持有序参考图；选中有效视频档案时，系统先提取精确任务帧或真实任务片段。Codex 逐镜头显示时间、目的、可见动作、运行目录内的精确参考素材、来源标识、连续性和风险；用户可以删减、重排、改意图或换素材，修改必须实际落入分镜并重算总时长。

#### Acceptance criteria

- [ ] 分镜不再使用固定五秒时长或固定镜头角色序列。
- [ ] 每个镜头包含精确时间范围、观众侧目的、可见动作、节奏/转场意图、连续性说明和风险。
- [ ] Codex 逐镜头显示原始图片路径和图片本身，不要求打开分镜文件。
- [ ] 删除、重排、改写镜头目的和替换图片会更新真实分镜产物，而不只是记录意见。
- [ ] 每次修改后重新编号、重新计算时间轴和预计总时长，并重新显示受影响镜头。
- [ ] 替换图必须来自允许的官方图片或通过现有受控覆盖校验，路径缺失和风险素材明确阻断。
- [ ] 分镜修改不会改写产品身份、正式事实、平台、语言或已确认策划的受保护边界。
- [ ] 分镜确认前不会生成 Dreamina Prompt 或 Dreamina task plan。
- [ ] 外部行为测试覆盖动态镜头数、展示内容、删除、重排、语义修改、换图和时长重算。

### 7. S07 — 将可视化分镜与标准 SRT 锁定为一个确认合同

- **类型**：AFK
- **Blocked by**：S06
- **覆盖用户故事**：46–52

#### What to build

交付分镜加 SRT 的共同确认路径：生成分镜时同步生成标准 `.srt` 草稿，字幕内容只来自观众侧表达和正式事实，并显式映射到单镜头或连续同义镜头组；允许有意静默。Codex 同时打印完整分镜核心信息和 SRT cue，用户一次确认后共同锁定。

#### Acceptance criteria

- [ ] 分镜草稿生成时同时产生符合标准序号、时间码和空行规则的 `.srt` 文件。
- [ ] 每条 cue 记录其对应镜头或连续同义镜头组，时间范围与当前时间轴一致。
- [ ] 字幕语义必须与镜头目的、可见动作和正式产品事实一致。
- [ ] 画面描述和 Dreamina Prompt 永远不能作为字幕缺失时的自动回退文本。
- [ ] 允许镜头明确标记 intentional silence，且不会机械生成空泛字幕。
- [ ] SRT 是未来旁白唯一逐字稿，不再生成内容冲突的独立 voiceover script 或 Markdown 字幕稿。
- [ ] Codex 共同确认视图逐镜头显示时间、目的、动作、图片、连续性、风险以及对应 cue 或有意静默。
- [ ] 普通“确认”只在共同确认视图有效时锁定分镜和 SRT。
- [ ] 修改镜头目的、时间、顺序或删除镜头会重生成受影响 cue，并撤销原共同确认。
- [ ] 字幕语义不匹配、时间码异常或映射断裂会阻止确认并明确指出镜头。
- [ ] 测试覆盖单镜头 cue、镜头组 cue、有意静默、语义风险、时间重算和确认失效。

### 8. S08 — 仅在分镜与 SRT 确认后生成 Prompt 和即梦任务计划

- **类型**：AFK
- **Blocked by**：S07
- **覆盖用户故事**：53–56、58

#### What to build

交付从已确认创意合同到单图 Dreamina 任务计划的完整路径：共同确认后才内部生成机器可读与审计用 Prompt，再按镜头生成任务。Codex 直接显示每个任务的镜头映射、参考图、时长、预计额度、适配信息和 blocker，不要求用户单独确认 Prompt 文案。

#### Acceptance criteria

- [ ] 分镜与 SRT 未共同确认时，不存在新的 Prompt artifact 或 Dreamina task plan。
- [ ] 确认后自动生成机器输入 Prompt 和审计用 Prompt，二者均可追溯到已确认镜头。
- [ ] Prompt 使用目标视频语言，不混入中文操作者标签或可读字幕文字。
- [ ] 每个任务映射到一个确认镜头、参考图、时长和预期输出。
- [ ] 任务创建前校验所选模式的时长、分辨率、文件类型、图片数量和其他可验证能力。
- [ ] Codex 任务确认视图逐项显示镜头、任务类型、参考图、时长、预计额度、适配和 blocker。
- [ ] 用户无需打开 Prompt 或任务文件即可理解将要付费生成的内容。
- [ ] Prompt 不设独立用户确认门禁；用户确认的是镜头执行合同和随后显示的付费任务计划。
- [ ] Prompt 或任务计划中的未知字段、路径丢失、时长冲突和声明风险明确阻断。
- [ ] 测试覆盖产物时序、目标语言、任务追溯、额度展示、能力门禁和 Codex 确认视图。

### 9. S09 — 支持有序多图镜头并适配真实 Dreamina 能力

- **类型**：HITL
- **Blocked by**：S08
- **覆盖用户故事**：40–43、55–57

#### What to build

把单图镜头纵向扩展为 Shot Reference Sequence：同一连续镜头可以使用多张有序参考图，每张图有开场、过程、细节、动作或收束等明确角色，并完成产品身份、尺度、环境、光线、动作和转场连续性检查。任务适配器只有在真实模式支持时才把多图传入同一任务；不支持时提出连续拆镜方案并回到分镜加 SRT 共同确认。

#### Acceptance criteria

- [ ] 镜头数据、Codex 展示、Prompt 和 Dreamina 任务均保留完整有序参考序列及每图角色。
- [ ] 每张参考图都已经完成像素检查并在分镜确认视图中显示。
- [ ] 多图镜头记录产品身份、尺度、环境、光线、动作和转场的连续性结论。
- [ ] 偶然跨镜头重复或镜头内无意义重复会阻止确认；明确批准的有意重复可追溯。
- [ ] 真实 Dreamina 模式支持多图时，一个连续用户镜头映射为一个任务，命令和记录不丢失顺序。
- [ ] 真实模式不支持多图时，不丢图、不静默取首图、不暗中拆任务。
- [ ] 不支持时在 Codex 中提出连续镜头修订，显示新时间轴和 SRT 映射，并等待重新确认。
- [ ] 兼容读取旧单图镜头时必须显式归一化为长度为一的参考序列，不能把旧字段继续作为新主合同。
- [ ] 测试覆盖支持、不支持、超上限、顺序保持、连续性 blocker、重复批准和拆镜回退。
- [ ] HITL 验收使用实际选定 Dreamina 模式验证多图 CLI/API 合同，并由用户确认采用“单任务多图”或“拆镜重确认”的真实能力结论。

### 10. S10 — 建立即梦付费计划确认与真实提交双门禁

- **类型**：AFK
- **Blocked by**：S09
- **覆盖用户故事**：59–62、66

#### What to build

交付安全的付费执行路径：用户审阅 Codex 中完整任务计划后，只有明确说出“确认即梦生成”才能批准额度；批准后仍需独立的“提交即梦任务”动作才会形成真实提交或安全手动交接。保留已有可靠 CLI、PowerShell、断点恢复和 submit ID 记录协议。

#### Acceptance criteria

- [ ] 普通“确认”、确认策划或确认分镜均不能授权 Dreamina 额度。
- [ ] 只有明确的“确认即梦生成”在无 blocker 的当前任务计划上生效。
- [ ] 额度确认只锁定任务计划，不会自动提交任何外部任务。
- [ ] 真实提交需要独立且明确的提交动作，提交前再次校验计划哈希、参考图和授权状态。
- [ ] dry-run、mock、blocked 和真实提交在 Codex 与持久状态中有不可混淆的标识。
- [ ] 现有 CLI 参数、PowerShell 手动交接、断点恢复、真实 submit ID 和任务映射协议保持可用。
- [ ] Codex 直接打印操作者需要的安全交接内容；用户不需要读取内部 JSON 才能执行。
- [ ] 任务计划在额度确认后发生变化时，原授权自动失效。
- [ ] 测试覆盖普通确认拒绝、显式额度确认、独立提交、计划变更失效、dry-run 和提交记录。

### 11. S11 — 用网页端完成声明触发一次性下载、校验和合并

- **类型**：AFK
- **Blocked by**：S10
- **覆盖用户故事**：63–69

#### What to build

交付生成后的简化闭环：用户在 Dreamina 网页端自行监测，系统不再暴露自动状态查询步骤。用户声明全部生成完成后，运行一次 One-Shot Dreamina Download，按真实 submit ID 获取全部镜头并验证数量、映射和可读性；验证成功后按确认分镜顺序合并，并保留锁定 SRT 作为后续字幕/旁白交接。

#### Acceptance criteria

- [ ] 自然语言帮助、Skill 和运行时不再把“查询即梦结果”作为用户步骤或下一步建议。
- [ ] 只有已记录真实 submit ID 且用户声明网页端全部完成时，才能触发一次性下载。
- [ ] 单次动作按任务计划获取所有预期结果，不实现持续轮询或隐藏重试循环。
- [ ] 下载后校验任务 ID 到镜头映射、预期文件数、文件存在性、非空性和媒体可读性。
- [ ] 缺失、mock、dry-run、损坏或无法映射的文件会阻止合并并明确报告。
- [ ] 验证成功后按已确认故事板顺序合并镜头，不使用旧状态中的顺序。
- [ ] 合并交付保留已锁定 `.srt`，不另写冲突 voiceover script，不烧字幕、不加 BGM。
- [ ] 用户拒绝生成结果时记录未接受并停止当前运行，不静默回退、重试或重新提交。
- [ ] 不满意结果的下一步是新建视频任务，当前版本不暴露结果改进流程。
- [ ] 测试覆盖无 submit ID、未完成声明、一次下载、文件校验、成功合并和拒绝终止。

### 12. S12 — 为旧运行提供可恢复读取或明确迁移失败

- **类型**：AFK
- **Blocked by**：S11
- **覆盖用户故事**：70–71、74

#### What to build

交付跨版本恢复路径：继续已有运行时先读取真实状态，识别旧 TikTok 平台、固定六问、单图字段、提前 Prompt、Markdown 字幕、状态查询和镜头重试等旧合同。仅在语义不变且不会绕过新门禁时安全归一化；否则提供明确迁移失败和新建任务建议。

#### Acceptance criteria

- [ ] 恢复运行始终以持久状态为准，不根据对话记忆猜测阶段。
- [ ] 每次恢复回执显示当前阶段、已确认合同、遗留 warning/blocker 和唯一安全下一步。
- [ ] 旧单图字段可在含义明确时归一化为单项 Shot Reference Sequence。
- [ ] 旧固定访谈答案只有能映射到新语义决策且证据完整时才迁移；否则重新进入对应未决项。
- [ ] 旧 TikTok 或双平台运行不会被静默改成已确认的 YouTube-only 运行。
- [ ] 旧提前生成 Prompt 不会被视为当前已确认 Prompt，必须遵守新的分镜加 SRT 门禁。
- [ ] 旧 Markdown 字幕、状态查询完成标记、重试计划和镜头确认不能绕过标准 SRT、一次性下载或新任务规则。
- [ ] 无法安全迁移时状态明确 blocked，保留原文件并建议创建新任务，不破坏旧审计记录。
- [ ] 测试使用代表性旧运行夹具覆盖安全归一化、重新追问和明确失败三类结果。
