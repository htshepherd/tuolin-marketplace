# `tuolin-linkedin-search` 关键词驱动版：纵向切片开发文档

**日期：** 2026-07-23
**来源 PRD：** `docs/prd/tuolin-linkedin-search-keyword-driven.md`
**目标版本：** `1.53.0`
**发布边界：** 本地开发文档，不创建 GitHub Issue
**执行约定：** 用户已一次确认 S01–S11 的粒度、依赖及 AFK/HITL 分类；开发按依赖顺序连续推进，不重复询问已确认的业务决策。

## 1. 切片原则

每个切片必须交付一条可运行、可观察、可持久化并可自动验证的端到端路径，而不是只修改一个 schema、函数、Skill 文案或测试夹具。源码、插件镜像、自然语言行为、持久化产物和测试必须在同一切片内保持一致。

- **AFK**：可使用确定性状态和浏览器观察夹具完成实现与自动验证。
- **HITL**：必须由人在真实 Codex Chrome 与 LinkedIn 环境确认可见结果或授权外部动作。
- S10 只读验收不得发送邀请。
- S11 真实邀请必须在届时显示精确候选批次后获得最终外部动作授权；当前开发授权不能替代该门禁。

## 2. 依赖关系

```text
S01 纯关键词运行入口
 └─ S02 六项访谈与运行简报
     └─ S03 账号绑定与首次 Posts 搜索合同
         ├─ S04 个人贴文完整候选卡
         └─ S05 可验证无限滚动
              \
S04 + S05 ───── S06 有序组合关键词回退
     └───────── S07 同行/渠道/下游/未解析线索
S06 + S07 ───── S08 人工审核、无找补与关键词留言
                   └─ S09 新 schema 下的发送、账本与恢复
                       └─ S10 真实 Chrome 只读验收（HITL）
                           └─ S11 单条真实邀请验收（HITL）
```

S04、S05、S07 在各自前置完成后可以独立实现；本轮按用户要求依次推进。

## 4. 详细切片

### S01 — 创建纯关键词搜索任务

**类型：** AFK
**Blocked by：** None - can start immediately
**用户故事：** 1–10

#### What to build

从自然语言路由到持久化运行目录交付一条无产品、无知识接口依赖的新任务路径。运行 schema 不再包含产品或产品上下文标识，但必须保留可恢复状态、运行简报入口、共享账本位置和明确的 keyword-driven 标记。测试阶段不迁移旧产品绑定运行。

#### Acceptance criteria

- [ ] 没有产品卡、Agent 接口或知识上下文时可以创建新运行。
- [ ] 运行目录、requirements、workflow state 和 change log 可恢复且不包含产品字段。
- [ ] 运行创建只校验可写任务存储，不读取知识库。
- [ ] 自然语言路由仍与发帖策划、配图、审阅和发布技能隔离。
- [ ] 旧产品绑定运行不会被静默迁移到新 schema。
- [ ] 源码、插件镜像、CLI 和外部行为测试同步。

### S02 — 完成六项访谈与关键词运行简报

**类型：** AFK
**Blocked by：** S01
**用户故事：** 11–37

#### What to build

交付从新运行到已确认运行简报的完整路径。关键词题只给格式指导并等待用户输入；其余缺失字段保持一个推荐、一个理由、当前问题确认。解析任意数量的逗号、中文逗号、分号、中文分号或换行分隔的完整短语。

#### Acceptance criteria

- [ ] 访谈只包含关键词、排序、日期、留言、间隔和最大邀请数六项。
- [ ] 初始请求已有答案不重复询问。
- [ ] 关键词题不生成推荐词；有效自定义答案只保存一次并立即进入下一题。
- [ ] 多词短语保持完整；列表按输入顺序、大小写不敏感去重并保留首次写法。
- [ ] 不拆词、翻译、纠错、扩词、拼接 Boolean 查询或限制关键词数量。
- [ ] 简报显示去重后的数量、完整顺序和 exact source phrase。
- [ ] `确认` 仅接受当前推荐，不支持一次确认所有剩余问题。
- [ ] 默认 Latest、Past month、最多 10 人和固定 5 分钟建议保持有效。

### S03 — 绑定账号并记录首次 Posts 搜索

**类型：** AFK
**Blocked by：** S02
**用户故事：** 38–44

#### What to build

使用确定性 Chrome 观察合同完成已登录账号绑定、只读授权、容量计算和第一个 exact phrase 的 Posts 搜索记录。该切片不依赖真实浏览器即可通过夹具演示完整状态转换。

#### Acceptance criteria

- [ ] 显示姓名和 profile URL 的账号身份被绑定到唯一运行。
- [ ] 未登录、未知身份或账号变化明确阻断。
- [ ] 只记录官方 Codex Chrome surface，不保存认证秘密。
- [ ] 首次搜索严格匹配 exact phrase、Posts、排序和日期范围。
- [ ] 运行状态初始化唯一贴文集合、打开计数和滚动证据。
- [ ] 请求上限与滚动容量计算仍能产生有效运行上限。

### S04 — 从个人贴文生成完整候选卡

**类型：** AFK
**Blocked by：** S03
**用户故事：** 58、64–69、72–77

#### What to build

从一条个人作者贴文观察贯穿初筛、身份校验、Connect 门禁、共享去重、candidate reservation、Codex 输出及 Markdown/JSON 持久化。卡片打印完整可见正文，不以摘要替代。

#### Acceptance criteria

- [ ] 明显噪声产生带理由的 Skip。
- [ ] 合理但不确定的商业相关性标记为 provisional，而不是确认客户。
- [ ] 个人 profile、公司、Connect 路径和实时连接状态通过门禁后才能形成卡片。
- [ ] 卡片包含 exact source phrase、完整贴文正文/URL、作者/公司、联系人、支持证据和疑点。
- [ ] Codex、Markdown 与 JSON 内容一致。
- [ ] sent、pending、connected、reserved 或 ambiguous 联系人不会形成新卡片。

### S05 — 实现可验证的无限滚动耗尽

**类型：** AFK
**Blocked by：** S03
**用户故事：** 45–54

#### What to build

增加确定性的无限滚动证据状态。每次滚动记录前后唯一贴文身份数量、是否等待加载完成以及连续无增长次数。只有候选上限、50 条唯一已审核贴文或连续 3 次无增长才允许结束关键词。

#### Acceptance criteria

- [ ] 广告、placeholder 和重复 URL/URN 不计入 50 条。
- [ ] 新唯一贴文出现时连续无增长计数归零。
- [ ] 一次或两次无增长不能声明耗尽；第三次才可以。
- [ ] 页脚、当前可见数量和被移除的 `page=2` 参数不能成为耗尽证据。
- [ ] below-limit 的提前 `finish-keyword` 被确定性拒绝。
- [ ] 状态和证据可在中断后恢复并继续计数。

### S06 — 实现有序组合关键词回退

**类型：** AFK
**Blocked by：** S04、S05
**用户故事：** 25–37、55–57

#### What to build

把单关键词路径扩展为 ordered fallback progression。当前短语达到合法停止条件但候选不足时进入下一短语；候选池达到有效上限时结束发现。每条产物保持 source phrase。

#### Acceptance criteria

- [ ] 短语严格按简报顺序独立搜索。
- [ ] 当前短语未合法结束时不能切换下一项。
- [ ] 候选不足时从 verified exhaustion 或 50 条上限进入下一项。
- [ ] 候选池达到有效上限后不搜索剩余短语。
- [ ] 所有短语结束时允许零候选或不足上限的正常结果。
- [ ] 不自动扩词、重复、放宽筛选或切换 People。

### S07 — 区分同行、渠道商、下游用户与未解析线索

**类型：** AFK
**Blocked by：** S04
**用户故事：** 59–72

#### What to build

建立窄接口的 provisional classification contract，并覆盖个人与公司作者分支。明确自产同类基础材料者 Skip；渠道品牌和下游使用方可进入人工判断；相关公司无法解析可连接个人账号时保存 unresolved lead 并继续搜索。

#### Acceptance criteria

- [ ] 同类基础材料制造商/直接供应商以可见证据判为 competitor Skip。
- [ ] 安装商、承包商、集成商、制品制造商和设备商无需公开 RFQ 即可 provisional Continue。
- [ ] 品牌商、经销商、零售商和贴牌方可 provisional Continue，并披露 make-versus-source 不确定性。
- [ ] 无法解析个人 profile 或 Connect 路径的相关公司记录 unresolved lead，不占候选名额。
- [ ] 公司作者最多选择一名符合现有角色优先级的联系人。
- [ ] 六条真实审核示例作为回归夹具覆盖供应商、安装商、误命中、制品商、软件语义和销售经理。

### S08 — 完成人工候选审核、无找补和关键词留言

**类型：** AFK
**Blocked by：** S06、S07
**用户故事：** 73–88

#### What to build

交付完整候选批次在 Codex 中的人工审核路径。卡片显示完整贴文和 provisional 证据；用户删除后不重新发现。启用留言时只根据关键词共同主题生成克制英文文本，没有可靠共同主题时使用通用行业联系措辞。

#### Acceptance criteria

- [ ] Codex 批量输出每张完整候选卡而不是仅给摘要。
- [ ] 人工删除释放适用 reservation，但不重新滚动或找补。
- [ ] 最终批准批次允许少于 10 人并冻结为 closed batch。
- [ ] 留言不访问知识库，不包含产品参数、认证或公司能力声明。
- [ ] 留言必须由用户确认或编辑后冻结，批次执行期间逐字复用。
- [ ] Add a note 不可用时仍执行现有重新授权门禁。

### S09 — 让发送、账本和恢复适配新运行格式

**类型：** AFK
**Blocked by：** S08
**用户故事：** 89–98

#### What to build

在不依赖 product/context 字段的新 schema 上回归完整授权、预检、顺序发送、账本记账、滚动容量、局部失败、平台停止和中断恢复。打包 1.53.0 并保持源/插件镜像一致。

#### Acceptance criteria

- [ ] 最终授权显示账号、精确候选、数量、note、间隔和有效上限，不引用产品。
- [ ] 默认 10 人和账号范围 100/168h 本地成功记录边界保持有效。
- [ ] 每条发送前重新校验账号、卡片、实时状态、账本和间隔。
- [ ] 成功可见后原子记账；局部失败继续；限制/CAPTCHA/安全检查/退出登录停止全批。
- [ ] 普通中断按原 closed batch 对账和重新授权；平台停止不自动恢复。
- [ ] 全量自动测试、Skill 校验、插件校验、镜像校验和打包校验通过。

### S10 — 安装新版并完成真实 Chrome 只读验收

**类型：** HITL
**Blocked by：** S09
**用户故事：** 38–83

#### What to build

在老板电脑安装 1.53.0，新开 Codex 会话，从关键词创建新运行，绑定真实账号并完成只读 Posts 搜索和候选审核，不发送邀请。

#### Acceptance criteria

- [ ] 无知识接口和产品选择仍能创建运行并完成六项访谈。
- [ ] 真实 Chrome 账号与搜索条件正确显示。
- [ ] `exhaust wrap` 搜索不会在 6 条、广告或页脚处提前结束。
- [ ] 滚动继续到候选上限、50 条唯一贴文或三次无增长证据。
- [ ] 至少一张可行动候选卡完整打印贴文和联系人证据，或真实记录 verified exhaustion/unresolved lead。
- [ ] 整个切片不打开或提交连接邀请。

### S11 — 执行一条真实连接邀请验收

**类型：** HITL
**Blocked by：** S10
**用户故事：** 83–98

#### What to build

老板审核精确候选卡后，对一个候选完成最终批次授权和一次真实连接邀请，验证浏览器可见成功、运行状态、候选卡结果和共享账本一致。

#### Acceptance criteria

- [ ] 发送前再次显示账号、完整候选卡、note/no-note、间隔和有效容量。
- [ ] 当前操作人对精确候选明确授权后才允许点击。
- [ ] LinkedIn 可见确认发送成功后才计为 success。
- [ ] 成功立即进入候选卡、运行报告和账号共享账本。
- [ ] 歧义结果不重试；平台告警立即停止。
- [ ] 验收不扩大到第二名候选或无人值守批量发送。

## 5. 完成定义

- S01–S09 全部实现并通过自动化验证；
- S10 在安装后的真实 Chrome 环境完成只读验收；
- S11 仅在精确候选获得最终外部动作授权后完成一次真实邀请验收；
- PRD、CONTEXT、ADR、Skill、运行时、插件镜像、安装文档和版本信息一致；
- 未完成的 HITL 必须明确标为 pending，不能以 fake browser 或 dry-run 冒充真实验收。
