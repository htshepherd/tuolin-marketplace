## 1. 切片原则

本文把 `$tuolin-linkedin-search` 拆成 14 个 Tracer Bullet 纵向切片。每个切片必须交付一条可运行、可验证的用户路径，不能只完成 schema、脚本、Skill 文案、浏览器适配器或测试中的某一层。

每个切片在其适用范围内都必须贯穿：

- 自然语言入口与 Skill 指令；
- 产品知识或前序运行状态；
- 确定性业务策略；
- Codex 中的用户可见结果与确认门禁；
- Markdown/JSON 等持久化产物；
- 源码、插件镜像和安装后的行为一致性；
- 只验证外部行为与持久化合同的自动测试。

稳定边界：

- 每个任务只绑定一个经验证的拓霖正式产品；
- 产品知识只从 Agent 读取接口获得，不扫描 `raw`，不读取领英发帖活动产物；
- 用户自定义市场词只用于搜索，不得提升为正式产品名、参数或声明；
- 领英操作只通过 Codex Chrome 官方插件和用户现有登录状态执行；
- Codex 不接收、保存或处理密码、OTP、Cookie、local storage 等会话秘密；
- 候选人和运行记录不进入正式知识库；
- 所有外部连接邀请必须绑定明确账号、封闭候选批次和最终人工授权；
- 不实现 headless Chrome、CDP 规避、浏览器指纹伪装或“不封号”保证；
- 成功标准是领英页面确认邀请已发出，不包含邀请接受率或销售转化率。

## 2. 类型说明

- **AFK**：需求、边界和验收标准已经明确，可由开发 Agent 独立实现、测试和提交。
- **HITL**：自动化实现完成后，仍必须由人使用真实 Codex Chrome/LinkedIn 环境完成验收或授权外部动作。

## 5. 详细切片

### S01 — 建立产品绑定的搜索任务

**类型：** AFK
**Blocked by：** 无，可立即开始
**覆盖用户故事：** 1–5、85–88、94–95

#### What to build

交付最小可演示任务入口：用户用自然语言提出基于某个拓霖产品搜索 LinkedIn 潜在客户后，系统路由到独立 `$tuolin-linkedin-search` Skill，创建隔离运行目录，从已验证 Agent 读取接口生成专用 `linkedin_search` 上下文，并把正式产品身份、允许的英文名称、应用场景和销售用途保存到初始状态。产品缺失、非正式、接口过期或产品冲突时必须阻断并引导到 `$tuolin-kb`，不得进入访谈。

#### Acceptance criteria

- [ ] 自然语言中的“领英搜索客户/潜客/通过贴文找客户”等意图进入新 Skill，不会误入现有发帖策划、配图、营销审阅或发布流程。
- [ ] 每次请求创建独立运行目录和可恢复状态，不复用或污染旧任务。
- [ ] 初始状态只绑定一个 canonical product ID，并记录 Agent 接口修订和上下文生成时间。
- [ ] `linkedin_search` 上下文只包含正式产品、应用场景和销售材料允许字段，且 `raw_access=false`。
- [ ] 用户输入的关键词、帖子内容或候选信息不会写回正式知识库。
- [ ] 产品不存在、卡片非 official、接口缺失/过期、产品冲突时，运行明确标为 blocked，并给出 `$tuolin-kb` 的自然语言下一步。
- [ ] 现有 `$tuolin-linkedin` 仍只负责内容与发帖工作，不新增隐式搜索或连接动作。
- [ ] Codex 回执显示当前阶段、绑定产品、知识修订、运行目录、阻塞项和下一步。
- [ ] 源 Skill、插件镜像、路由和外部行为测试保持一致。

#### Verification

- 使用临时项目分别验证正式产品、缺失产品、过期接口和冲突产品。
- 验证运行产物不包含 `raw` 路径、活动产物或候选人信息。
- 验证现有 LinkedIn 和视频自然语言测试无回归。

### S02 — 完成逐题需求访谈与运行简报

**类型：** AFK
**Blocked by：** S01
**覆盖用户故事：** 6–19、61、66、68–69

#### What to build

交付从产品绑定状态到 Confirmed Search Brief 的完整路径。访谈组件从初始请求提取已知答案，只对会改变可支持搜索条件、候选边界或外部动作的缺失决策逐题追问。每轮显示一个编号中文问题、一个推荐答案和理由，并以“是否确认？”结束；普通“确认”只确认当前项。

#### Acceptance criteria

- [ ] 已在初始请求明确的产品、关键词、排序、时间范围、留言、间隔或数量不会重复询问。
- [ ] AI 推荐关键词默认使用英文，并可追溯到正式产品上下文。
- [ ] 用户自定义 `exhaust wrap` 等市场词以 `search_only` 保存，不会进入正式名称或产品声明。
- [ ] 关键词列表保留用户确认的顺序。
- [ ] 缺少排序时推荐 `Latest`；缺少发布日期时推荐 `Past month`。
- [ ] 不询问 Posts 搜索无法应用的地区条件。
- [ ] 内容类型、发布者、提及对象、作者公司/行业/职位等可选条件仅在相关且当前 UI 支持时询问。
- [ ] 每轮只有一个问题、一个推荐、一个理由和一个“是否确认？”。
- [ ] 普通“确认”只确认当前问题，不接受剩余建议；不支持“一次确认所有剩余问题”。
- [ ] 默认运行邀请上限为 10；数量是上限而不是必须完成的配额。
- [ ] 留言和固定发送间隔作为后续外部动作决策进入简报；间隔推荐值为五分钟。
- [ ] 所有必需决策明确后生成 Confirmed Search Brief，并显示已确认值、来源和仍存在的 warning。
- [ ] 未满足访谈完整性时不能进入浏览器搜索阶段。

#### Verification

- 用纯数据测试初始请求字段提取、逐题顺序、当前项确认、用户修正和状态恢复。
- 验证不支持的地区问题和无业务影响的问题不会出现。
- 验证 search-only 词不会改变产品上下文或邀请留言中的正式事实。

### S03 — 绑定 LinkedIn 账号并完成只读贴文搜索

**类型：** HITL
**Blocked by：** S02
**覆盖用户故事：** 55–58、89–93

#### What to build

交付第一条真实浏览器 tracer：Confirmed Search Brief 完成后，通过 Codex Chrome 官方插件在专用任务标签组中打开 LinkedIn，读取当前已登录会员的姓名和 profile URL，绑定账号，然后应用第一个已确认关键词、Posts 类型、排序和发布日期条件。该切片只读取搜索结果，不打开连接邀请写操作。

#### Acceptance criteria

- [ ] 只使用 Codex Chrome 官方插件，不启动仓库自带 headless Chrome，也不切换到内置 in-app Browser。
- [ ] 任务使用独立且可识别的 Chrome 标签组。
- [ ] 浏览器适配器读取并返回当前会员姓名、profile URL 和可验证的登录状态。
- [ ] 未登录、账号身份未知或身份与运行绑定不一致时立即 blocked。
- [ ] Codex 不请求密码、OTP，不把 Cookie、local storage 或会话数据写入日志和运行产物。
- [ ] 搜索明确选择 Posts，并应用已确认关键词、排序和发布日期条件。
- [ ] 不尝试应用未受支持的地区条件。
- [ ] 适配器优先使用结构化 DOM 交互，必要时才使用截图/视觉回退。
- [ ] 运行状态记录账号公开身份和搜索条件，但不记录认证秘密。
- [ ] Codex 显示绑定账号、已应用条件、可见结果数量和下一步，不发送邀请。

#### Verification

- 自动合同测试使用可控浏览器观察夹具验证已登录、未登录、账号变化和筛选条件。
- HITL 使用真实已登录 Chrome 完成一次只读搜索，并人工确认账号和筛选条件正确。
- 检查运行目录，确认不存在认证秘密或默认截图。

### S04 — 从一条贴文生成个人作者候选卡

**类型：** AFK
**Blocked by：** S03
**覆盖用户故事：** 20、26–32、37–38、45–50

#### What to build

交付从一条可见搜索结果到候选卡的完整路径。系统打开一条贴文，提取可见正文、URL、作者和公司信息，由 Codex 给出 `Continue` 或 `Skip` 及一个理由。个人作者代表相关品牌、分销商或零售商且存在标准 Connect 路径时，生成并打印一张候选卡，同时保存匹配的 Markdown 和 JSON。

#### Acceptance criteria

- [ ] 每条打开的贴文必须产生 `Continue` 或 `Skip`，并附一个基于可见内容的简短理由。
- [ ] 销售、推广、分销、品牌、采购、转售或贴牌目标品类的内容可以 Continue。
- [ ] 新闻转发、同名词误命中、普通消费者内容和垃圾内容明确 Skip。
- [ ] 同品类品牌、分销商或零售商在可能采购、转售或贴牌时可标记为 Same-Category Channel Prospect。
- [ ] 个人作者已体现相关商业主体时直接作为候选，不自动替换成更高职级同事。
- [ ] 只有存在标准 Connect 路径时才能形成可发送候选；需要邮箱或额外身份信息时 Skip。
- [ ] 不使用 Follow、Message、InMail 或外部邮箱作为替代动作。
- [ ] 候选卡包含关键词、贴文正文/URL、相关理由、作者/公司、姓名、职位、公司和 profile URL。
- [ ] 候选卡同时写为 Markdown 和 JSON，字段语义一致，并直接打印在 Codex 中。
- [ ] 默认不保存截图；只有用户明确要求或交互状态有争议时保存截图证据。
- [ ] Skip 结果也记录最小审计信息和原因，但不进入候选批次。

#### Verification

- 使用个人作者、新闻转发、同名误命中、消费者内容、垃圾内容和渠道商夹具验证外部结果。
- 验证 Markdown/JSON 卡片内容一致且不包含认证秘密。
- 验证没有 Connect 路径或要求邮箱时不会生成可发送候选。

### S05 — 支持有序多关键词搜索与正常耗尽

**类型：** AFK
**Blocked by：** S04
**覆盖用户故事：** 21–25

#### What to build

把单关键词 tracer 扩展为完整的有序搜索循环。每个关键词最多打开检查 50 条贴文，然后进入下一个已确认关键词；达到有效候选上限或所有关键词耗尽时正常结束发现阶段。系统不得为了凑够人数自动扩词、放宽筛选、重复搜索或回补。

#### Acceptance criteria

- [ ] 关键词严格按照 Confirmed Search Brief 的顺序执行。
- [ ] 每个关键词单独记录已扫描结果、已打开贴文和生成候选数量。
- [ ] 单关键词最多打开 50 条贴文；该内部上限不会在业务访谈中询问。
- [ ] 达到当前有效运行上限时停止继续发现。
- [ ] 当前关键词耗尽或达到检查上限时进入下一个关键词，不重复当前搜索。
- [ ] 所有关键词耗尽时以实际候选数量正常完成发现阶段，零候选也是合法结果。
- [ ] 不自动生成新关键词、不放宽排序或时间范围、不切换到 People 搜索。
- [ ] 不为被跳过、去重或被用户删除的候选自动找补。
- [ ] Codex 进度回执显示关键词位置、检查数量、候选数量、跳过数量和停止原因。

#### Verification

- 用确定性分页夹具验证关键词顺序、50 条边界、提前达到候选上限和正常耗尽。
- 验证零候选、少于请求上限和全部关键词结束都不会触发隐式扩搜。
- 验证恢复时不会重复已经完成的关键词或贴文。

### S06 — 从公司贴文解析唯一联系人

**类型：** AFK
**Blocked by：** S04
**覆盖用户故事：** 29、33–38

#### What to build

交付公司作者贴文的端到端候选路径。相关贴文由公司主页发布时，系统在该公司中寻找一个存在标准 Connect 路径的人员，并按照 Owner/Founder、Procurement/Purchasing/Sourcing、Managing Director/General Manager、Product Manager 的优先级选择。找不到合格人员时明确 Skip。

#### Acceptance criteria

- [ ] 公司贴文通过与个人贴文相同的相关性判断后才进入联系人解析。
- [ ] 角色优先级严格按已确认顺序应用，同一优先级使用稳定、可解释的选择规则。
- [ ] 每家公司最多产生一个候选联系人。
- [ ] 联系人卡保留原公司贴文、公司主页和所选个人 profile 的可追溯关系。
- [ ] 未找到目标职能、身份不明确或没有标准 Connect 路径时明确 Skip。
- [ ] 不猜测职位、不要求邮箱、不使用 Follow、Message 或 InMail 替代。
- [ ] Same-Category Channel Prospect 仍必须满足潜在采购、转售或贴牌的商业解释。
- [ ] Codex 候选卡显示公司为何相关、为何选择此人及其角色优先级。

#### Verification

- 使用覆盖四级角色优先级、同级多人、无合格角色和无 Connect 路径的公司夹具。
- 验证同一公司不会产生多个候选。
- 验证选择结果在 Markdown、JSON 和 Codex 展示中一致。

### S07 — 建立跨任务联系人账本和实时去重

**类型：** AFK
**Blocked by：** S05、S06
**覆盖用户故事：** 35–44、76

#### What to build

交付账号范围的 Prospecting Contact Ledger、Active Run Lock 和两次对账路径。候选卡展示前，系统使用规范化的个人、公司和贴文 URL 查询账本并读取领英当前连接状态；发送前再次执行同样检查。已经发送、待处理、已连接、被其他未完成批次保留或状态不明的联系人不会进入或继续执行。

#### Acceptance criteria

- [ ] 账本按绑定的 LinkedIn 账号公开身份隔离，不同账号不会共用状态。
- [ ] profile、company 和 post URL 经过确定性规范化，常见查询参数、尾部斜杠和等价 URL 不会绕过去重。
- [ ] 已发送、pending、connected、reserved 或 ambiguous 状态在候选卡展示前被过滤。
- [ ] 手动在 LinkedIn 发出的邀请可通过实时状态识别并抑制，即使本地账本没有记录。
- [ ] 一家公司成功联系后，其其他员工在后续任务中被过滤。
- [ ] 只有能够确定未发出的失败记录才允许重试；状态不明不能重试。
- [ ] 候选进入待审核批次时建立可追溯 reservation，批次结束或明确放弃时按规则释放。
- [ ] 同一账号同一时间只允许一个活动运行；锁冲突明确 blocked。
- [ ] 候选卡展示前和每次发送前都执行本地加实时的双重对账。
- [ ] LinkedIn 已确认成功但账本写入失败时停止运行并标记 reconciliation required。
- [ ] 账本和锁使用原子写入，异常不会留下半条成功记录或静默覆盖其他运行。

#### Verification

- 临时目录测试跨任务个人/公司去重、账号隔离、URL 规范化、reservation 和锁冲突。
- 浏览器夹具测试 pending、connected、手动发送和 ambiguous 状态。
- 注入账本写入失败，验证页面成功后运行立即停止且不得继续下一人。

### S08 — 完成批量候选审核与封闭批次

**类型：** AFK
**Blocked by：** S07
**覆盖用户故事：** 45–54

#### What to build

交付候选发现到 Closed Candidate Batch 的完整人工审核路径。发现结束后，Codex 按批次打印全部候选卡及来源证据；用户可以自然语言删除候选。最终确认把剩余名单冻结为封闭批次，随后不能自动增加、替换或重新发现候选。

#### Acceptance criteria

- [ ] 候选批次只包含通过展示前去重和实时状态检查的候选。
- [ ] Codex 为每名候选显示关键词、贴文、相关理由、作者/公司、联系人身份和 profile URL。
- [ ] Markdown 和 JSON 候选卡与 Codex 展示具有相同的业务字段和稳定 candidate ID。
- [ ] 用户可以按序号、姓名或 profile URL 删除候选，删除结果写入运行状态和审计记录。
- [ ] 删除候选后不自动回到搜索阶段，也不从已跳过结果中补人。
- [ ] 确认前再次显示剩余精确名单和数量。
- [ ] 普通确认只在当前批次确认问题上生效，不授权发送。
- [ ] 批次确认后生成不可扩充的 Closed Candidate Batch，并记录内容摘要/哈希。
- [ ] 批次冻结后新增、替换、重排或修改 candidate identity 会使后续授权无效并被拒绝。
- [ ] 需要继续发现时必须新建任务，不能重开当前封闭批次。
- [ ] 默认不生成截图；争议证据截图与对应候选和原因绑定。

#### Verification

- 测试展示、按多种标识删除、零人批次、确认和冻结后的修改拒绝。
- 验证删除后无 backfill，确认后无新搜索。
- 验证 Markdown、JSON、Codex 视图和批次哈希一致。

### S09 — 完成留言确认和最终发送授权

**类型：** AFK
**Blocked by：** S08
**覆盖用户故事：** 55–66

#### What to build

交付从封闭批次到 Authorized Dispatch Batch 的完整门禁。若用户选择留言，AI 基于正式产品和本次搜索目的生成一条简短、克制、不夸大的英文 Add a note 文案，用户确认或修改后逐字冻结。发送前重新读取账号身份和有效上限，并在 Codex 中显示账号、精确候选、数量、留言/无留言、固定间隔和上限计算；只有该视图上的明确确认才授权执行。

#### Acceptance criteria

- [ ] 留言为可选决策；未选择留言时明确记录 `no_note`，而不是空字符串推断。
- [ ] AI 留言默认英文，简短、非夸大、不过度推销，且只使用正式产品上下文允许的信息。
- [ ] 用户可确认或修改留言；最终文本以精确字节值和内容摘要冻结。
- [ ] 批次执行期间不得按候选人自动改写、翻译或个性化留言。
- [ ] 最终授权前重新读取当前 LinkedIn 姓名和 profile URL。
- [ ] 账号缺失、未知或不同于运行绑定账号时立即 blocked，不能沿用旧授权。
- [ ] 最终简报列出每名候选、绑定账号、批次数量、留言精确文本或无留言、固定间隔、请求上限和有效上限。
- [ ] 批次内容、留言、账号、间隔或上限变化都会使旧授权失效。
- [ ] 用户在最终简报上的明确确认生成 Authorized Dispatch Batch；无需逐人重复确认。
- [ ] 如果运行时 Add a note 不可用或平台额度不足，必须暂停并重新询问改为无留言或结束，不得静默移除留言。

#### Verification

- 测试留言生成边界、用户修改、逐字冻结和批次执行期间不可改写。
- 测试账号变化、批次变化、间隔变化和上限变化使授权失效。
- 测试 Add a note 不可用时只允许重新授权无留言或结束。

### S10 — 发送一条连接邀请并可靠记账

**类型：** HITL
**Blocked by：** S09
**覆盖用户故事：** 37、40、43–44、60、64–65、77、89–93、96

#### What to build

交付最窄的真实外部动作 tracer：对 Authorized Dispatch Batch 中的一名候选，在发送前重新校验账号、账本和 LinkedIn 当前状态，使用已确认的留言或无留言路径点击 Connect。只有 LinkedIn UI 明确显示邀请已发出时才产生 Invitation Dispatch Success，并立即原子写入账本和运行报告。

#### Acceptance criteria

- [ ] 没有有效最终授权时，任何自然语言“继续”“确认”或恢复动作都不能发送邀请。
- [ ] 发送前重新校验绑定账号、candidate identity、Connect 路径、本地账本和实时状态。
- [ ] pending、connected、ambiguous、无 Connect 或要求邮箱时不发送。
- [ ] 有留言批次只能发送冻结的精确文本；无留言批次不会打开或填写 Add a note。
- [ ] 只有 LinkedIn UI 明确确认邀请已发出时才记录 Invitation Dispatch Success。
- [ ] 点击后状态不清楚时记录 ambiguous 并停止，不重复点击。
- [ ] 成功事件保存账号、候选、公司、源贴文、批次、时间和结果，不保存认证秘密。
- [ ] 成功后先完成账本原子写入，再允许状态进入下一个候选。
- [ ] 页面成功而账本失败时运行进入 reconciliation required，不得继续。
- [ ] Codex 清楚显示实际结果是成功、跳过、失败还是状态不明。
- [ ] 功能不宣称规避检测、平台批准或账号安全保证。

#### Verification

- 自动合同测试使用假浏览器覆盖成功、pending、connected、无 Connect、要求邮箱、状态不明和账本失败。
- HITL 在真实 LinkedIn 中由用户审阅精确候选并明确授权后，只发送一条邀请。
- 人工核对 LinkedIn 页面状态、运行报告和共享账本三者一致。

### S11 — 扩展为顺序发送、间隔和滚动上限

**类型：** AFK
**Blocked by：** S10
**覆盖用户故事：** 66–77

#### What to build

把单条邀请 tracer 扩展为整批顺序执行。系统使用用户确认的固定间隔，默认运行上限 10，并根据当前账号过去滚动 168 小时内由本 Skill 记录的成功数计算剩余容量；有效运行上限等于请求上限和剩余容量的较小值。容量在发现前和每次发送前重新计算。

#### Acceptance criteria

- [ ] 未指定运行上限时使用 10，且明确显示这是 ceiling 而不是 quota。
- [ ] 每个绑定账号在任意滚动 168 小时内最多记录 100 次由本 Skill 完成的 Invitation Dispatch Success。
- [ ] 手工邀请或其他工具的动作不计入本地上限，并在运行简报中明确说明这一局限。
- [ ] 有效运行上限为 `min(requested_limit, remaining_recorded_capacity)`。
- [ ] 有效上限低于用户请求时，在浏览器发现前显示计算并要求确认。
- [ ] 剩余容量为零时在发现前 blocked，不打开搜索流程。
- [ ] 每次实际发送前重新计算滚动容量；达到零时停止剩余批次。
- [ ] 邀请严格顺序执行，上一人的结果和账本持久化完成后才处理下一人。
- [ ] 连续邀请之间使用用户确认的固定时间间隔，默认推荐五分钟。
- [ ] 不引入随机抖动，不把间隔描述为“模拟真人”或规避检测。
- [ ] Active Run Lock 覆盖整个账号执行窗口，防止并行运行突破上限。
- [ ] 完成结果可以少于请求上限，不自动回填或扩大搜索。

#### Verification

- 使用可控时钟测试恰好 168 小时边界、跨时区时间、100 次边界、请求值小于/大于剩余容量。
- 测试发送前重新计算、固定间隔和严格顺序。
- 测试两个同账号运行竞争锁与不同账号独立容量。

### S12 — 处理候选失败和平台级停止

**类型：** AFK
**Blocked by：** S11
**覆盖用户故事：** 78–82

#### What to build

交付批次执行中的错误分类和停止报告。仅影响一个候选的确定性错误记录后可以继续；LinkedIn 限制、安全检查、CAPTCHA、退出登录或连续无法解释的错误属于 Platform-Level Stop，必须立即停止整个批次并保存未执行候选。

#### Acceptance criteria

- [ ] 候选局部失败具有明确分类，例如按钮消失、候选已变为 pending 或 profile 不可访问。
- [ ] 只有能够确定没有发送的局部失败才可记录为 retryable；状态不明不可重试。
- [ ] 候选局部失败写入报告后可以继续下一名已授权候选。
- [ ] 限制提示、安全检查、CAPTCHA、退出登录或重复未知错误被识别为 Platform-Level Stop。
- [ ] Platform-Level Stop 发生后不再点击、导航到下一候选或等待后自动恢复。
- [ ] 当前运行标记为 incomplete/stopped，并保留未执行候选和停止证据。
- [ ] Codex 报告分别列出 succeeded、skipped、failed、ambiguous 和 unexecuted。
- [ ] 报告指出是否存在 reconciliation required，以及下一步必须新建任务还是允许普通恢复。
- [ ] 截图仅在争议或平台级停止证据需要时保存，并避免包含不必要的会话数据。

#### Verification

- 使用浏览器夹具逐类验证局部失败、限制、安全检查、CAPTCHA、退出登录和未知错误阈值。
- 验证平台级停止后的浏览器动作计数为零。
- 验证报告分类数量与持久化候选状态完全一致。

### S13 — 支持中断恢复和授权重启

**类型：** AFK
**Blocked by：** S12
**覆盖用户故事：** 81–84、86

#### What to build

交付两条明确分开的恢复路径。普通 Codex、Chrome 或网络中断可以继续同一运行，但必须读取真实状态、重新验证账号、对账最后一个外部动作、重新显示剩余封闭批次并获得新的最终发送授权；Platform-Level Stop 不允许原运行自动恢复，剩余候选只能在新任务中形成 Authorized Dispatch Restart。

#### Acceptance criteria

- [ ] 继续任务时首先读取运行状态和账本，不依赖对话记忆猜测阶段。
- [ ] 普通中断恢复前重新绑定相同账号并检查最后一个候选的实时 LinkedIn 状态。
- [ ] 最后动作状态不明时进入 reconciliation required，不重复发送。
- [ ] 普通恢复只显示原 Closed Candidate Batch 中尚未执行的候选。
- [ ] 恢复不得重新搜索、增加候选、替换候选或为失败人员找补。
- [ ] 继续发送前重新计算滚动容量、重新显示间隔/留言和剩余批次，并获得新的最终授权。
- [ ] Platform-Level Stop 的原运行保持 incomplete，不能通过“继续”恢复。
- [ ] Platform-Level Stop 后必须创建新任务，并以新的账号和批次授权形成 Authorized Dispatch Restart。
- [ ] 新任务引用原运行和保留候选，但仍执行账本、实时状态、容量和账号检查。
- [ ] stale lock 只能在状态和外部动作完成对账后释放，不能按超时盲删。
- [ ] Codex 恢复回执显示原运行、当前阶段、最后动作、对账结论、剩余候选和唯一安全下一步。

#### Verification

- 在访谈、发现、批次确认、点击后未记账和发送间隔中断等不同阶段注入普通中断。
- 验证恢复始终需要账号复核、最后动作对账和新授权。
- 验证 Platform-Level Stop 无法原地恢复，而新任务会再次执行所有门禁。

## 6. 统一完成定义

单个切片只有同时满足以下条件才算完成：

- 可通过自然语言触发并产生用户可理解的 Codex 回执；
- 贯穿所需知识、状态、浏览器合同或外部动作，而不是孤立内部模块；
- 生成可恢复、可审计且不包含认证秘密的持久化产物；
- 自动测试验证外部行为、门禁和失败状态，不依赖私有 helper 的具体实现；
- 源码与插件分发镜像同步；
- 不破坏既有 `$tuolin-linkedin`、知识库和视频工作流边界；
- HITL 切片必须明确记录真实验收结果，不能用 mock、dry-run 或自动测试替代。

## 7. 推荐开发顺序

1. 依次完成 S01–S04，尽早打通“自然语言请求 → 产品上下文 → 真实只读搜索 → 一张候选卡”的第一条 tracer。
2. 并行开发 S05 和 S06，然后通过 S07 收敛到统一的账号范围账本。
3. 完成 S08–S10，形成“候选审核 → 最终授权 → 一条真实邀请”的第二条 tracer。
4. 完成 S11–S13，把单条路径扩展为受限批次、失败停止和安全恢复。
5. 最后执行 S14 的安装态全量回归和两次独立 HITL 验收。
