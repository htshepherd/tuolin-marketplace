# LinkedIn 应用层 Agent 切片开发文档

**来源 PRD：** `doc/PRD-2.0.md`  
**PRD 版本：** v2.0 draft-4  
**日期：** 2026-06-17  
**目标：** 将 LinkedIn 30 天发帖活动需求拆成可独立开发、可独立验收、能端到端跑通的纵向切片。  
**目标仓库：** `tuolin-marketplace`，在同一个 marketplace 中新增 LinkedIn 应用层 Agent，不拆新仓库、不新增独立安装包。

## 1. 切片原则

本文件只覆盖 LinkedIn 应用层 Agent，不重新拆分知识库 2.0 主线。每个切片都必须跑通一条可验证的业务路径，不能只做“数据层”“脚本层”“文档层”的横向工作。

切片类型：

- **AFK**：可以由开发者独立完成，不需要业务人员中途判断。
- **HITL**：必须有人工确认、业务判断或验收输入。

所有切片必须遵守以下边界：

- LinkedIn Agent 是应用层 Agent，只通过 Agent读取接口使用知识。
- 不直接扫描 raw。
- 不写回 `knowledge/okf/`。
- 不把内容生产结果当作正式知识。
- 不自动发布 LinkedIn，不做账号授权、排程或数据分析。
- 产品内部检索仍使用“石英纤维隔热带”，对外发布使用“特种玻璃纤维带 / Specialty Glass Fiber Tape”。
- 内容素材卡只能用于配图、画面说明和配图 brief，不能证明耐温、安全、无烟、不刺痒等产品事实。

## 4. 详细切片

### 01. LinkedIn Agent 入口与知识库项目校验闭环

**类型：** AFK  
**Blocked by：** 无  
**覆盖 PRD：** 8.11、UC-010

**要实现什么：**

- 新增 LinkedIn 应用层 Agent 的 Codex skill 入口。
- 支持用户用中文自然语言触发 LinkedIn 30 天活动需求。
- 运行时默认把当前工作目录当作知识库项目目录。
- 校验当前目录是否具备知识库项目必要结构和 Agent读取接口。
- 当前目录无效时，用业务语言提示用户切换到知识库项目目录或先初始化知识库。
- 不把内部脚本包装成普通员工直接使用的独立 CLI 产品。
- 保持同一个 `tuolin-marketplace` 安装包，不拆新仓库。

**验收标准：**

- 用户输入 LinkedIn 30 天宣传需求时，系统能识别为 LinkedIn 活动任务。
- 当前目录缺少有效知识库结构时，系统停止并给出清晰提示。
- 当前目录有效时，系统进入 LinkedIn 活动流程，不扫描 raw。
- 新 skill 安装后不影响现有知识库 skill。
- 普通员工不需要记忆脚本名称或参数。

**测试建议：**

- 构造有效知识库项目目录，验证入口可以进入下一步。
- 构造缺少 `generated/agent-interface` 的目录，验证提示明确。
- 构造普通插件仓库目录，验证不会误当成知识库项目。

**风险与注意：**

- marketplace 仓库不是企业知识库数据目录本身，入口必须区分插件目录和知识库项目目录。
- 第一版不要引入全局固定路径，避免员工在不同项目目录之间混淆。

### 02. LinkedIn 任务上下文与素材边界闭环

**类型：** AFK  
**Blocked by：** 01  
**覆盖 PRD：** 3.13、7.2、UC-010

**要实现什么：**

- 扩展 `linkedin_post` 任务上下文，使其读取产品卡、应用场景卡、销售物料卡和内容素材卡。
- 只读取 `official` 且外部使用范围允许的内容。
- 内容素材卡只能用于配图、画面说明和配图 brief。
- 产品事实、性能卖点、合规承诺仍必须来自产品卡、应用场景卡、销售物料卡、证据或本次人工确认卖点。
- 上下文中显式写明 `raw_access: false` 和素材使用边界。
- 上下文生成后可写入 `generated/agent-interface/contexts/`，但不得写回正式知识层。

**验收标准：**

- `linkedin_post` 上下文包含可用内容素材卡。
- `customer_question` 等内部客服内容不会默认进入 LinkedIn 对外上下文。
- 内容素材不会被标记为性能事实证据。
- 上下文策略说明 LinkedIn Agent 不得扫描 raw。
- 相关知识更新后，旧上下文按现有机制失效并可重建。

**测试建议：**

- 复用现有 downstream context 测试风格。
- 构造 official/external_allowed 内容素材，验证被选入上下文。
- 构造 draft、review_required、internal_only 内容，验证不会作为对外素材使用。
- 验证 `content_asset` 只进入素材集合，不进入事实证明集合。

**风险与注意：**

- 这个切片不能扩大为“直接读取图片文件夹”。素材必须从 Agent读取接口来。
- 不能为了配图方便突破“图片不能证明产品事实”的规则。

### 03. 中文策划生成与桌面活动文件夹闭环

**类型：** AFK + HITL 验收  
**Blocked by：** 01、02  
**覆盖 PRD：** 8.11、UC-010

**要实现什么：**

- 从用户自然语言和 LinkedIn 任务上下文生成中文整体策划。
- 在桌面创建活动文件夹，命名为 `拓霖领英30天_特种玻璃纤维带_YYYYMMDD_HHMM`。
- 写入 `01_中文策划.md`。
- 初始化活动目录结构：
  - `daily/`
  - `assets/logo/`
  - `assets/source-images/`
  - `assets/publishing-images/`
- 初始化 `campaign-manifest.json`，记录活动状态为等待策划确认。
- 策划内容应覆盖目标受众、主题节奏、核心卖点、内容栏目、配图方向和人工确认点。

**验收标准：**

- 桌面生成活动文件夹。
- `01_中文策划.md` 存在且可读。
- 策划面向欧美工业采购、工业品分销商、维修维护团队和热管理相关从业者。
- 策划中对外中文产品名为“特种玻璃纤维带”。
- 系统生成策划后停止，等待用户确认，不直接生成 30 天贴文。
- manifest 记录策划文件路径、生成时间、产品映射和当前阶段。

**测试建议：**

- 使用临时 Desktop 目录测试文件夹命名和文件写入。
- 使用固定 fixture 上下文测试策划内容包含必要栏目。
- 验证未确认前不会生成 `02_中文30天贴文总稿.md`。

**风险与注意：**

- 这里不是生成最终发布稿，只是策划。
- 不要把策划内容写入知识库，也不要刷新正式知识层。

### 04. 策划确认到中文30天贴文总稿闭环

**类型：** AFK + HITL 验收  
**Blocked by：** 03  
**覆盖 PRD：** 8.11、UC-010

**要实现什么：**

- 识别用户对策划的确认意图。
- 从已确认策划和 LinkedIn 上下文生成 `02_中文30天贴文总稿.md`。
- 中文总稿包含 Day 1 到 Day 30 的每日主题、中文贴文正文、配图方向、画面 tags 草案和需要复核的卖点提示。
- 使用“特种玻璃纤维带”作为对外产品名。
- 保留内部产品映射，但不在用户发布内容中暴露内部英文名。
- 生成后更新 manifest 状态为等待中文总稿确认。

**验收标准：**

- 生成一个 Markdown 中文总稿，而不是 30 个分散中文文件。
- 总稿包含完整 Day 1 到 Day 30。
- 每天内容围绕同一产品但主题不重复堆叠。
- 重点卖点包含耐高温1000度、不刺痒、不冒烟，但缺少正式证据时标记为本次人工确认卖点。
- 生成中文总稿后停止，等待用户确认，不直接生成英文发布包。

**测试建议：**

- 验证 Day 数量严格为 30。
- 验证中文发布内容不出现“石英纤维”作为对外产品名。
- 验证 manifest 从 `planning_ready` 进入 `chinese_draft_ready`。

**风险与注意：**

- 中文总稿是内部审稿版本，不是正式知识。
- 不要在这个切片里引入英文翻译或图片合成。

### 05. 中文总稿确认到英文每日发布包闭环

**类型：** AFK + HITL 验收  
**Blocked by：** 04  
**覆盖 PRD：** 8.11、UC-010

**要实现什么：**

- 识别用户对中文总稿的确认意图。
- 生成英文发布总览 `03_英文发布总览.md`。
- 生成英文发布日历 `04_英文发布日历.csv`。
- 生成 `daily/day-01.md` 到 `daily/day-30.md`。
- 每个每日文件包含：
  - Day 编号
  - 英文标题或主题
  - 120-180 词英文 LinkedIn 正文
  - 3-5 个专业 LinkedIn hashtags
  - 轻量 CTA
  - 推荐配图或配图 brief
  - 发布前复核提示
- 排期只使用 Day 1 到 Day 30，不绑定具体日期。
- 适合询盘、索样或索取规格时使用 `tuolin@tuolintech.com`。

**验收标准：**

- 生成 30 个每日英文发布文件。
- 生成英文总览和 CSV 日历。
- 英文发布内容不出现 `Quartz Fiber Tape`。
- 英文默认使用 `Specialty Glass Fiber Tape`。
- 每帖包含 3-5 个专业 hashtags。
- 系统不调用 LinkedIn 发布 API，不做账号授权或排程。

**测试建议：**

- 验证每日文件数量和命名。
- 验证 CSV 行数和 Day 编号。
- 验证每帖词数在合理范围内。
- 验证禁止词 `Quartz Fiber Tape` 不出现。

**风险与注意：**

- 翻译不是逐字翻译，应面向 LinkedIn B2B 发布场景改写。
- CTA 要克制，不要每帖都硬塞邮箱。

### 06. 对外命名、卖点证据与发布安全闭环

**类型：** AFK  
**Blocked by：** 03、04、05  
**覆盖 PRD：** 术语表、3.13、UC-010、11

**要实现什么：**

- 建立内部产品名与对外产品名映射：
  - 内部：石英纤维隔热带 / `product/quartz_fiber_tape`
  - 对外中文：特种玻璃纤维带
  - 对外英文：`Specialty Glass Fiber Tape`
- 检查所有发布产物，禁止对外内容出现 `Quartz Fiber Tape`。
- 对“耐高温1000度、不刺痒、不冒烟”执行卖点来源标记：
  - 有正式外部知识支撑时，记录为正式知识卖点。
  - 知识库缺少但用户任务明确给出时，记录为本次人工确认卖点。
- 本次人工确认卖点必须在 manifest 和发布包中提示发布前复核。
- 确保 LinkedIn Agent 不写回正式知识层。

**验收标准：**

- 所有用户可复制发布稿使用对外产品名。
- manifest 记录产品映射和卖点来源。
- 缺证据卖点不会被伪装成正式知识。
- 本次人工确认卖点不会反向写入 `knowledge/okf/`。
- 发布包中有清晰的人工复核提示。

**测试建议：**

- 构造知识库有证据与无证据两种 fixture。
- 验证卖点来源标记不同。
- 验证对外禁用名称扫描。
- 验证不会修改正式知识目录。

**风险与注意：**

- 这个切片是发布安全底线，不能只靠提示词约束。
- 对外命名规则应集中实现，避免不同生成阶段各写各的。

### 07. 配图素材选择与图片 brief 闭环

**类型：** AFK  
**Blocked by：** 02、03  
**覆盖 PRD：** 7.2、8.11、UC-010

**要实现什么：**

- 从 LinkedIn 上下文里的内容素材卡选择可对外使用的真实素材。
- 真实素材优先用于每日配图建议。
- 没有可用真实素材时，生成配图 brief 和 AI 作图提示。
- 配图 brief 必须说明：
  - 每日主题
  - 推荐画面
  - 不应伪装成产品实拍
  - 建议画面 tags
  - 是否需要人工补充图片
- 选择素材时不得扫描 raw 文件夹。

**验收标准：**

- 有可用内容素材时，每日文件关联素材建议。
- 无可用内容素材时，每日文件包含配图 brief 和 AI 作图提示。
- brief 中不会把 AI 图称为真实产品照片。
- 画面 tags 根据每日主题生成，不从固定配置硬编码。
- 内容素材仍不作为性能事实证据。

**测试建议：**

- 构造有图片素材和无图片素材两种上下文。
- 验证生成结果分支不同。
- 验证 brief 包含“不伪装成实拍”的约束。

**风险与注意：**

- 第一版可以先做到素材选择和 brief，不必在同一切片完成图片合成。
- 不要为了找到图片去读 raw 目录。

### 08. 透明 logo、水印 tags 与发布图生成闭环

**类型：** AFK + HITL 验收  
**Blocked by：** 07  
**覆盖 PRD：** 7.2、8.11、UC-010

**要实现什么：**

- 支持用户提供独立透明 logo 文件。
- 缺少透明 logo 时，给出清晰提示，不生成低质量水印图。
- 对可用真实素材生成发布图，输出到 `assets/publishing-images/`。
- 发布图叠加：
  - Tuolin logo
  - 2-3 个每日主题画面 tags
- 画面 tags 与 LinkedIn hashtags 分开处理。
- 参考图只作为版式样例，不自动裁切 logo。

**验收标准：**

- 缺少 logo 时流程可停止在可理解提示，不崩溃。
- 提供透明 logo 和测试图片时，可以生成非空发布图。
- 发布图位于活动文件夹内。
- 每日文件能引用对应发布图路径。
- 画面 tags 不写死在配置里，而是来自每日主题。

**测试建议：**

- 使用测试透明 logo 和测试产品图生成发布图。
- 验证输出图片存在且非空。
- 验证没有 logo 时不会生成发布图。
- 验证 manifest 记录 logo 文件和发布图状态。

**风险与注意：**

- 如果引入图片处理依赖，应保持轻量，并在 Windows 下可安装。
- 不要从用户参考图硬裁 logo，避免背景和边缘污染。

### 09. 活动 manifest、三段确认与恢复闭环

**类型：** AFK  
**Blocked by：** 03、04、05、07、08  
**覆盖 PRD：** 8.11、UC-010、11

**要实现什么：**

- 用 `campaign-manifest.json` 记录活动状态和关键产物。
- 状态至少包括：
  - `planning_ready`
  - `planning_confirmed`
  - `chinese_draft_ready`
  - `chinese_draft_confirmed`
  - `english_package_ready`
  - `image_assets_ready`
- 记录产品映射、卖点来源、活动文件夹、logo 路径、每日文件、发布图、上下文 ID。
- 支持用户在同一活动文件夹中继续下一阶段。
- 重复执行同阶段时避免静默覆盖用户已编辑文件；需要生成新版本或提示用户。
- 三段确认必须明确：
  - 先确认中文策划
  - 再确认中文总稿
  - 最后生成英文发布包

**验收标准：**

- 每一阶段完成后 manifest 状态正确。
- 未确认策划时不能生成中文总稿。
- 未确认中文总稿时不能生成英文发布包。
- 重新打开活动文件夹后可以继续下一阶段。
- 用户手动修改过的文件不会被静默覆盖。

**测试建议：**

- 模拟完整三段状态流转。
- 模拟跳过确认直接请求下一阶段，验证阻止。
- 模拟已存在输出文件，验证不会静默覆盖。

**风险与注意：**

- 三段确认是业务流程，不只是 UI 文案。
- manifest 是活动状态，不是正式知识。

### 10. 自然语言入口、文档与全链路验收闭环

**要实现什么：**

**验收标准：**
